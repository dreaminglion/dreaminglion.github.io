### 项目一：合规检测相关项目

**合规事前管控**
1. 双周稳定性、合规宣导，季度合规考核，将合规要求、评估、设计等理念在客户端研发团体达成共识。
2. 制定需求评估合规部分，协助安全、法务团队，在需求设计阶段给出合规评估与建议，提早与产品达成共识，确认客户端编码涉及合规的函数、用户数据等进行评估。
3. 服务台解答，编辑整理国标合规文件、内外部合规案例，对内部同学的问题进行解答。

**合规开发管控**
在App提测、灰度前进行对应的静态检测，动态检测，各手机品牌的合规检测，对版本新增Diff变更进行确认，并在系统中进行授权，如发现异常，进行对应的管控 如：添加放量开关、使用基础加密组件、使用AOP降低系统调用，补充合规披露等方式达到业务与合规的需求。

**静态检测项目**
扫描隐私函数、敏感词、函数调用链，按版本对比，在网页上展示版本增量，按条目进行授权备案的管理项目。
1. 隐私函数检测：以配置的隐私函数，在 smali（鸿蒙ark_disasm） 中遍历对应调用函数的名称、类，确认一级调用位置。组成Json对象进行版本之间的 Diff 增量对比。
2. 敏感词检测：读取 dex 文件的 string_ids 段，按偏移量与长度 new String 得到字符串内容。通过正则表达式顾虑中文的，进行版本diff 增量对比。
   - 鸿蒙对 asm 文件逐行遍历 const-string 字符串定义进行提取。
3. Asm调用链：通过 bytex ClassGraph 字节码编译时的静态函数调用关系，生成树调用节点，遍历为对应的调用堆栈。
4. 三方SDK披露检测：通过反编译 dex 之后进行目录合并，然后 tree 获取SDK包名，完善SDK信息库。在版本检测时爬取App披露的名称、Apk当前版本上传的包名，进行匹配披露与App实际是否一致。

**动态检测项目**
按照SOP操作流程，启动 hook 拦截脚本，看这个流程中的隐私函数调用堆栈结构、频率进行管控。在自由编辑区对权限使用情况，异常总结记录的管理平台。
1. 通过 frida 拦截 hook 系统 java、C 函数，获取对应的调用堆栈（frida js 提供了C堆栈的获取），通过生成当前进程的 uuid 与App基本信息，上传后台落库。 前端基于特定的表格格式 + md 自由编辑区，生成版本的动态检测报告。
2. SDK 初始化调用治理，避免无需使用的 sdk 在启动阶段调用，按最小必要规则，在需要的场景合理进行调用。通过 dump 内存中加载的类，确认启动阶段那些三方SDK进行了加载。
3. 制定检测的 SOP 文档，检验隐私协议前，首页，权限使用，SDK使用等不同场景的隐私调用情况。

**隐私检测SDK**
使用 lsposed hook java 层，plt hook 系统 c 层（binder系统进程通信），js 注入拦截 H5 的请求。实现隐私函数的堆栈获取，网络请求的数据流抓取。以一个 SDK 的集成在App中，在开发测试阶段通过研发与业务测试，扩大合规的检测范围。

**合规事后管控**
1. 对外部反馈的问题进行复现，定位代码位置。如有开关及时关闭、严重时是否热修、发小版本及时修复，与法务、GR等团队保持沟通确认统一的执行方案。
2. 总结复盘事件根因，进行对应的后续优化方案，如：荣耀手机删除外部存储的文件，系统给出了通知提示，在知乎上出现了舆论。随即管控了 file.delete 函数，在对应手机进行验证后灰度，并给用户做出了解释。

### 项目二：架构（安全）相关项目
**KMS Apk 签名系统**
1. 独立开发的 Go（后端）、Vue（前端）、gradle 打包插件（打包签名）、apksinger（kms实现）签名系统。将App签名KS文件，上传系统后台，研发无接触实现对App打包签名。无需将这个Apk上传，仅对hash data实现 rsawithhash256 签名算法，数秒钟完成 V1V2V3 的签名。
2. 确认 V2 与 V3 的区别，对比 V3 证书轮转的细节差异，当选择证书时自动生成签名方案参数，支持非轮转，与多次证书轮转。
3. 确认google aab的签名上架流程，兼容对 abb 进行签名。

**App签名治理**
1. 自测时发现签名库多线程压测出现签名错误情况，排查 C AES 库使用的全局静态变量做的数据传递，改造为参数传递，支持多线程。
2. 签名排序参数内存快照，附加在request对象上，当出现签名错误时，上传对应的原始排序参数，与网管日志对比，实现全链路的日志排查。
3. 治理非预期签名错误，签名错误**单日uv降低200倍**。


**App环境检测**
1. 检测手机环境 Root（su）、模拟器、多开、hook（pid maps）、被调试（tracerPid）、签名等信息。帮助业务方提供的异常用户数据归因，Crash 分析提供参考。
2. 使用了 llvm 对核心 C 代码进行字符串加密，控制流平坦化、虚假控制流，并添加三方壳进行动态下发保护。

**单元测试平台**
当组件发版，自动执行一次单元测试，将测试结果飞书消息通知研发。如果错误，研发可以通过链接查看本次单侧的错误详情。
1. 跑通unitTest、androidTest解决项目与单侧框架的组件版本冲突，mock 框架的使用逻辑，让业务拿到文档可以上手写自己项目的单侧。
2. Jenkins 单侧执行脚本，打包机环境配置，安装模拟器，使 androidTest 可以顺利执行。
3. 开发 gradle 脚本解析单侧执行结果，错误详情，代码覆盖率，在平台进行展示。
